# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- Build: `cargo build`
- Test all: `cargo test`
- Test one module: `cargo test srp::` (or `crypto::`, `session::`, `api::`, `cli::`, `drive::`, `commands::`)
- Test one case: `cargo test srp::exchange_tests::srp_exchange_is_internally_consistent_against_real_modulus`
- Run: `cargo run -- login` / `cargo run -- logout` — requires a real terminal; password entry opens `/dev/tty` directly and fails without a controlling terminal (e.g. a non-interactive pipe).
- Run: `cargo run -- upload <local-file> <remote-folder>` — `<remote-folder>` is an existing `/my-files/...`-rooted path (upload does not create folders); the uploaded file keeps its local basename. No live Proton test account has been available during development — the entire upload path is unverified against a real server.

## Architecture

Single binary crate (`proton-drive`), no Cargo workspace, no async runtime — `ureq` for blocking HTTP. This is a deliberate choice, not an oversight: see "Global Constraints" in `docs/superpowers/plans/2026-07-19-proton-drive-cli-auth.md` before introducing a workspace split or an async runtime.

Module layout, bottom-up by dependency:
- `srp.rs` — pure math, no I/O. Proton's SRP-6a *variant*: server-signed modulus verification, a custom `expand_hash` primitive in place of every textbook `H()`, little-endian encoding throughout (including the modulus itself — this was a real bug caught during planning; big-endian parses as non-prime). Also the two distinct bcrypt-based password derivations — the SRP secret and the private-key passphrase use different salt handling and must not be conflated. The exact deviations from textbook SRP-6a are documented in the plan doc, not re-derived here.
- `crypto.rs` — OpenPGP key/message handling via the `pgp` crate. Covers both the auth milestone (unlocking a derived password against a private key, to validate it actually works) and upload's crypto: dual-subkey node-key generation (a signing primary plus an independently-passphrase-protected ECDH encryption subkey — a solo signing-only key cannot receive encrypted messages), per-file AES-256 content keys (generated once, reused across every revision, never regenerated), AEAD-gated block encryption/decryption, manifest signing, and extended-attributes (XAttr) building. `pgp` 0.20.0 depends on `rand 0.8` internally, trait-incompatible with the `rand 0.10` used elsewhere in this crate — the `rand08` dependency (`rand@0.8`, renamed via `package = "rand"`) exists for this. It lives in `[dependencies]`, not just `[dev-dependencies]`: upload's production code (`generate_node_key`, `encrypt_and_sign_block`, etc.) genuinely needs randomness, not just this module's tests.
- `api/` — `mod.rs` wraps `ureq`. Proton signals success/failure via a top-level JSON `Code` field independent of HTTP status, so `http_status_as_error(false)` is set and the code checks `Code` itself rather than relying on ureq's status-based errors. `auth.rs` holds the auth/keys/users request and response types; `account.rs` holds address and feature-flag fetching; `drive.rs` holds the Drive-specific wire types and calls (shares, folder children, link details, file/revision creation, the small-file combined-upload endpoints, block-upload preparation/verification, revision commit).
- `session.rs` — the only module that touches the OS keyring. Holds `Credentials`: session tokens plus the *derived* key passphrase (from `srp::compute_key_password`) — never the login password itself.
- `drive.rs` (top-level — distinct from `api::drive`) — Drive path resolution. There's no server-side "resolve path" endpoint; this walks a folder's children one path segment at a time from the `/my-files/...` root, decrypting each candidate's name to find a match.
- `commands/` — thin orchestrators (`login.rs`, `logout.rs`, `upload.rs`) wiring `srp`/`drive` → `api` → `crypto` → `session` together. No business logic of their own; `ApiClient` is currently a concrete type with no mock seam, so this layer has no automated test coverage beyond what's testable in isolation from a live client (a known gap).

`login` is hardcoded to auth version 4 and single-password mode — it errors clearly (not silently) if the server reports a different version or `PasswordMode != 1`. 2FA is explicitly out of scope for this milestone: Proton's own open-source reference SDK has no working call site for it to port from.

`upload` always creates a new revision on a same-name conflict — it never errors or prompts. It takes the small-file fast path (one combined API call instead of create/prepare/upload/commit) when `size * 1.1 < 131072` bytes *and* the `DriveSmallFileUpload` feature flag reads `true`; AEAD block framing is similarly gated behind `DriveCryptoEncryptBlocksWithPgpAead`. Any failure to positively confirm a feature flag as `true` (network error, unexpected shape, an explicit `false`) falls back to the safe/non-flagged path — there's no working reference implementation of Proton's feature-flag service in the SDKs to verify the wire shape against (same category of gap as 2FA, above). Thumbnail generation is out of scope; `ThumbnailList` is always sent empty.

Address selection (`commands::upload::run`) used to pick the first address with a primary+active key without cross-checking it against the share's actual `AddressID` — a gap carried forward from the auth milestone's own pattern. That's now fixed: it fetches the My Files share first and matches candidate addresses against the share's `AddressID` (`share.share.membership_address_id`) before picking a key, erroring clearly if none matches or the matched address has no active primary key. Inbound signature verification, previously an open gap, is now closed: `crypto::verify_signature`/`verify_signature_any`/`decrypt_and_verify_name`, paired with `drive::resolve_verifying_key`/`verify_claim`, verify node passphrases, names, and content-key packets against their signers. Passphrases and names use the exclusive model (a claimed signer if present, else the parent folder's key); content-key packets use `verify_signature_any`'s additive model instead — the node's own key (always a legitimate signer, since `generate_content_key` always signs with it) is checked unconditionally, with a resolvable claimed address key as an additional, not alternative, acceptable signer, and no parent-key fallback (the node key already covers the anonymous case). Verification is non-fatal everywhere: a failure warns to stderr and proceeds with the already-decrypted plaintext regardless, matching the reference SDK/CLI's own confirmed behavior — it never aborts a command. `download`, still not built, will reuse these same primitives when it exists, trusting the same node-key chain.

Download (fetching and decrypting Drive files) does not exist yet — that's a separate, not-yet-written plan.

For the exact protocol details (why each SRP deviation exists, which values are little-endian, the bcrypt salt construction, the block-upload protocol reference material) read `docs/superpowers/plans/2026-07-19-proton-drive-cli-auth.md` rather than re-deriving them — it was written and validated against Proton's real reference SDK and a live account.
